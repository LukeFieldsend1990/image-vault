1 · How the system fits together

Image Vault is a secure biometric-likeness archive. Talent (actors) store 3D scan packages of their likeness; industry users (production companies) license those scans for a specific production. Every actual file download requires dual-custody 2FA — both the talent and the licensee approve with their authenticator codes — and downloads run on time-limited tokens with a full audit trail.

Set expectations correctly. Image Vault is server-mediated, not "zero-knowledge" or "end-to-end encrypted". The platform handles the file bytes and can technically read stored content. Its security comes from access control — authentication, dual-custody 2FA, time-limited download tokens, audit logging — plus encryption at rest from the storage provider. Avoid describing it as zero-knowledge in any demo or copy.

The five roles

RoleWho they areWhat they do
talentActor / performer / modelUploads scans, approves/denies licences, approves downloads, can invite their own rep. Invite-only.
repAgent or agencyActs on behalf of managed talent (delegation). Invite-only.
industryProduction company / studio (the "licensee")Owns an organisation & productions, requests licences, initiates downloads. Can self-sign-up or be concierge-invited.
complianceUnion / regulator / insurerRead-only watcher. Never reaches file content.
adminYou & your partnerEverything. Whitelist-only — set in code, not assignable through the UI.

Note: industry is the current name for what older parts of the system call licensee — they are the same role.

Organisations vs. productions vs. packages vs. licences

  • Organisation — the company entity (a production company, VFX vendor, agency…). An industry user owns one and can run multiple productions under it.
  • Production — a specific film / TV show / game / commercial. Belongs to an organisation, has a cast list and default licence terms.
  • Package (scan) — a talent's uploaded 3D scan (mesh, textures, metadata such as scan type, resolution, polygon count). The thing being licensed.
  • Licence — the agreement linking one talent's package to one production: type, territory, dates, fee, AI-training permission. Moves through states (see reference).

2 · The alias-account trick

To test the full flow you need several separate accounts — at minimum one industry user and one talent — because dual-custody requires two different people to approve. You can run them all from the one inbox martin.davison@gmail.com using Gmail's plus-addressing: anything after a + is ignored for delivery but makes a distinct address.

# All of these deliver to martin.davison@gmail.com, # but each is a separate Image Vault account: martin.davison+studio@gmail.com → Industry user (production owner / licensee) martin.davison+talent@gmail.com → Talent (the actor being licensed) martin.davison+talent2@gmail.com → Second actor (optional) martin.davison+rep@gmail.com → Agent / rep (optional)
Each account is fully separate. Every alias needs its own authenticator (TOTP) entry — Image Vault makes 2FA mandatory. Add each as a distinct entry in your authenticator app and name it clearly (e.g. "IV — studio", "IV — talent") so you can tell the codes apart during the dual-custody download. Plus-addressing also means an alias counts as a "new" email for signup even though replies land in the same inbox.
2FA does not depend on email. The authenticator QR code is shown on screen during signup, so you can set up 2FA even if no invite email ever arrives. Email is only needed to receive the invite link and notifications — and as admin you can skip email entirely and copy the invite link directly (see Part C).

3 · Before you start

  • Confirm your partner's email is on the admin whitelist (lib/auth/adminEmails.ts). Admin is granted in code, never through the UI — if he isn't whitelisted, the /admin pages will be inaccessible.
  • Have an authenticator app ready (Google Authenticator, Authy, 1Password, or Bitwarden all work) with room for several entries.
  • Decide your alias plan (above). For a minimal end-to-end test you need two: +studio (industry) and +talent.
  • Use the production URL for the environment under test (e.g. https://changling.io). Paths below are written relative to that host.

Part A — Pre-build the production & invite the industry owner

The fastest way to onboard an industry user is the admin Concierge setup: you build the production first, then invite the owner so they arrive to a ready-made project.

1

Open the concierge page

As admin, go to /admin/productions/invite ("Set up & invite a production").

2

Fill in the company & invitee

  • Company name * — e.g. Davison Pictures
  • Invitee email * — martin.davison+studio@gmail.com  (this person becomes the organisation owner on signup)
3

(Optional) Link to TMDB & import cast

Search a real film/TV title under "Link to TMDB (optional)" to auto-fill the production name, type and year. If you link a title you can tick "Import the TMDB cast as reserved placeholders" to seed a cast list — handy for testing cast resolution later.

4

Set the production details

  • Production name * — e.g. Project Northwind
  • Type — Film / TV Series / TV Movie / Commercial / Game / Music Video / Other
  • Year, and optionally Union project number + SAG-AFTRA / Equity flags
5

(Optional) Set default licence terms

Tick "Set default licence terms" to pre-fill terms that flow onto cast invites: intended use, licence type (Digital double · Film / Game character / Commercial / AI avatar / Training data / Monitoring reference), territory, valid-from/to dates, proposed fee per actor (£), and Permit AI training.

6

Send it

Click Set up & send invite. You'll see "Production set up & invite sent" with a View production link. An invite email goes to the studio alias with a signup link valid 7 days.

What just happened under the hood: the concierge created a production company, an organisation (type production_company), the production, optional cast placeholders & default terms, and an industry invite tied to that org and production.

Part B — The industry user signs up

Now act as the studio alias. Open the invite email in martin.davison@gmail.com (it'll be the one addressed to +studio) and click "Set up your account", or paste the signup link.

1

Create the account

At /signup?invite=… the email and account type are pre-filled and locked (a green banner confirms "You've been invited as Industry"). Set a password (12+ characters), confirm it, and click Create account.

2

Set up two-factor authentication (mandatory)

You land on /setup-2fa ("Set up authenticator"). Scan the QR with your authenticator app — name the entry "IV — studio" — or click "Can't scan? Enter key manually" for the text key. Enter the 6-digit Verification code and click Verify and continue. You're now logged in.

3

Arrive at the production dashboard

Because the production was pre-built, you're routed to /productions and can open your ready-made project. (A self-signed-up industry user with no production instead gets the guided wizard at /productions/setup.)

Part C — Create a talent account

Talent is invite-only, so create one as admin. (Switch back to your admin login — keep the studio session in a separate browser/profile so you can flip between them.)

1

Send a talent invite

Go to /admin/invites ("Invites" → "Send Invitation"). Set:

  • Email — martin.davison+talent@gmail.com
  • Role — Talent

Click Send invite.

No email delivery? Use the link. Tick "Skip email notification (generate link only)" before sending — the page then shows a "Share this link directly" box with a Copy button. Paste that link straight into your other browser profile to sign up. The link is valid 7 days and pre-fills the email and role. (You can revoke any pending invite from the table below the form.)

2

Sign up as talent

Open the invite link as the talent alias. Same flow as Part B: set a 12+ char password → Create account → set up 2FA (name it "IV — talent", a separate authenticator entry) → Verify and continue.

3

Complete talent onboarding

Talent are routed to /onboarding, where they confirm their public identity (search TMDB to claim their acting profile). Finish it to reach the talent dashboard.

Part D — Talent uploads a scan package

Still acting as the talent alias:

1

Upload from the vault

From the talent dashboard (/dashboard), use the scan uploader to add a package. Files stream to storage and the package moves to ready once the processing pipeline validates it.

2

Fill in package metadata

Open the package under /vault/packages and set the descriptive fields — scan type (Light Stage / Photogrammetry / LiDAR / Structured Light), resolution, polygon count, compatible engines, tags. A licensee can request a licence against a ready package.

For a quick test any small placeholder file works — you're exercising the licence + dual-custody flow, not the scan quality.

Part E — Request a licence & approve it

1

Industry user requests the licence studio

As the studio alias, find the talent (e.g. via /directory) and start a licence request against their package. Work through the request form: licence type → project details (project name / production company / which org you're acting for) → commercial terms (territory, exclusivity) → AI & data terms → declaration. Submit. The licence is created in a pending state and the talent is notified.

2

Talent reviews & approves talent

As the talent alias, go to /vault/requests. You'll see the pending request (and any "Cast Invitation" / awaiting-package items). Open the details to review terms and fee, attach a package if needed, then Approve (or Deny with a reason). On approval the licence becomes approved and the licensee is notified.

Two states worth knowing: a request can start as AWAITING_PACKAGE (a cast slot created by the production before the talent has uploaded) which the talent converts to a real request, or as a direct PENDING request against an existing package. Either way the destination is APPROVED.

Part F — The dual-custody download

This is the headline security flow: an approved licence does not grant a silent download. Both parties must approve with live 2FA codes, and the resulting links expire. You'll be flipping between the studio and talent logins — this is exactly why each needs its own authenticator entry.

1

Licensee initiates studio

As the studio alias, open the approved licence under /licences and start the download. Enter your 6-digit authenticator code ("IV — studio") to verify. The request then waits on the talent — you'll see an "awaiting talent verification" state.

2

Talent authorises talent

As the talent alias, go to /vault/authorise (or the download-approval prompt on /vault/licences). Enter your code ("IV — talent"). Optionally set a pre-authorisation window so repeat downloads from this licensee skip your step for a set period (not available for AI-training licences).

3

Download studio

Once both codes are verified, time-limited download links are issued (valid ~48 hours). Back as the studio alias, download the file(s) — individually or as a zip. Every download is written to the audit log.

Codes are time-based. If you're slow flipping between the two logins, an authenticator code may roll over — just read the current one. The dual-custody session itself is valid for about an hour; if it lapses, re-initiate from step 1.

Gotchas & tips

  • Use two browser profiles (or one normal + one incognito). You'll constantly switch between the studio, talent, and admin sessions; separate profiles save you from logging in and out.
  • One authenticator entry per account. Don't reuse a single TOTP secret — each signup generates its own. Name them clearly; the dual-custody download needs two different codes back-to-back.
  • Plus-addressing = real, distinct accounts. +studio and +talent are different users to the system even though both inbox to martin.davison@gmail.com.
  • If invite emails don't arrive, use the admin "Skip email notification → generate link only" toggle and copy the link. 2FA never depends on email (QR is on-screen).
  • Passwords must be 12+ characters — the signup form enforces it.
  • Talent & rep are invite-only; industry can self-sign-up, but concierge pre-building gives the cleanest demo.
  • Admin is whitelist-only and cannot be granted through the UI — your partner must already be in lib/auth/adminEmails.ts.
  • Keep the "not zero-knowledge" framing when describing the platform to others.
